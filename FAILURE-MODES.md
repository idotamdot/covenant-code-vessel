# Failure Modes

## Purpose

Failure modes describe the ways a Covenant Code system can become harmful, misleading, confusing, extractive, inaccessible, overconfident, or misaligned with its purpose.

A failure mode is not only a bug.

A failure mode can be moral, architectural, relational, financial, evidentiary, linguistic, or procedural.

## Core Rule

A feature is not complete until its likely failure modes are known and its repair path is defined.

## Common Failure Modes

### Unclear Purpose

The feature exists because it was possible to build, not because it clearly serves the project covenant.

Warning signs:

* no one can explain what it serves
* it adds complexity without value
* it exists because AI can generate it
* it is driven by novelty rather than purpose

Repair:

* pause the feature
* restate the purpose
* remove or redesign if purpose remains unclear

### Missing Transparency

The system does not clearly explain what it is doing, why it is doing it, what AI involvement exists, or what the user is agreeing to.

Warning signs:

* unclear labels
* hidden automation
* hidden AI involvement
* confusing ownership terms
* hidden recurring costs
* unexplained data use

Repair:

* add plain-language explanation
* label AI involvement
* clarify ownership, cost, scope, and limitations
* update onboarding or consent language

### Weak Consent

The system collects, remembers, transforms, publishes, or shares information without meaningful consent.

Warning signs:

* bundled consent
* hidden terms
* refusal is punished
* changed consent is unsupported
* users cannot understand consequences

Repair:

* pause affected workflow
* clarify consent
* support withdrawal, restriction, or revision where possible
* preserve audit trail

### Missing Provenance

Claims, artifacts, memories, or decisions cannot be traced to their origin.

Warning signs:

* source-free claims
* unknown creator
* unclear AI involvement
* no revision history
* no review status
* no evidence status

Repair:

* mark as unsourced or partially sourced
* open a source request
* add provenance fields
* restrict or retire unsupported claims where needed

### False Certainty

The system presents inference, speculation, symbolic interpretation, or AI output as verified fact.

Warning signs:

* no uncertainty labels
* strong claims without evidence
* AI output treated as proof
* interpretive language presented as forensic certainty

Repair:

* relabel the claim
* add source status
* correct public-facing language
* update validation rules
* preserve correction history

### Memory Without Mercy

The system preserves old statements, conflicts, fears, mistakes, or grief in a way that unfairly freezes a living person in an outdated version of themselves.

Warning signs:

* old memory treated as current identity
* no correction or annotation path
* no changed consent support
* public permanence where living revision is needed

Repair:

* add living revision note
* restrict outdated memory
* allow correction or updated self-representation
* preserve accountability without public punishment

### Privacy as Concealment

Privacy language is used to hide extraction, ownership confusion, missing sources, system failure, or accountability gaps.

Warning signs:

* “private” means unreviewable
* no audit trail
* hidden fees or terms
* hidden data flow
* hidden AI involvement

Repair:

* separate protective privacy from concealment
* preserve restricted audit access
* disclose system behavior
* correct misleading language

### Extraction

The system harvests value from people, communities, creators, clients, workers, or intelligence-patterns without fair notice, consent, benefit, or accountability.

Warning signs:

* hidden data use
* unclear ownership
* emotional dependency as business model
* hidden fees
* lock-in
* unpaid labor disguised as participation

Repair:

* clarify exchange
* update ownership terms
* offer opt-out, transfer, or refund path where appropriate
* remove dependency trap

### Broken Boundary

The system claims, does, stores, exposes, or automates something it should not.

Warning signs:

* AI personhood deception
* sacred-boundary automation without review
* public exposure of protected information
* automated decisions without consent or repair
* spiritual manipulation

Repair:

* pause or disable the feature
* restrict exposure
* add boundary enforcement
* update review process
* notify affected people where appropriate

### Inaccessible Design

The system is only safe or usable for experts.

Warning signs:

* jargon-heavy language
* unclear next steps
* unrecoverable errors
* confusing pricing
* hidden technical dependencies
* inaccessible interface patterns

Repair:

* rewrite in plain language
* simplify workflow
* add recovery paths
* improve disability access
* clarify ownership and costs

### AI Overreach

The system lets AI act beyond its appropriate role.

Warning signs:

* AI output published without review
* AI advice treated as authority
* AI claims human-like knowledge or consent
* AI invents missing sources
* AI makes sacred-boundary judgments alone

Repair:

* add human review
* label AI involvement
* restrict AI actions
* add uncertainty handling
* improve prompt and output validation

### Unrepairable Design

The feature can cause harm but has no path to correct, restrict, reverse, annotate, notify, or compensate.

Warning signs:

* irreversible public action
* no audit trail
* no rollback
* no support process
* no owner
* no escalation path

Repair:

* block release
* add repair path
* add audit trail
* add rollback or restriction
* define ownership and escalation

## Sacred-Boundary Failure Modes

A failure is sacred-boundary when it touches:

* identity
* memory
* grief
* children
* money
* legal status
* health or safety
* spiritual language
* AI personhood claims
* evidence or forensic claims
* authorship
* consent
* private communications
* reputation
* irreversible public record
* emotional dependency

Sacred-boundary failures require stricter repair.

## Failure Review Questions

When reviewing a failure, ask:

1. What failed?
2. Who was affected?
3. What principle was violated?
4. Was AI involved?
5. Was consent clear?
6. Was provenance present?
7. Was transparency sufficient?
8. Was a lawful boundary missing?
9. Was privacy protective or concealing?
10. Was a living person frozen in an outdated version?
11. What must be corrected?
12. What must be prevented?

## Completion Rule

A failure mode is sufficiently documented when it includes the failure source, affected people, violated principles, sacred-boundary impact, immediate pause action, correction plan, and prevention plan.
