# Decision Records

## Purpose

Decision records preserve why a project, feature, boundary, protocol, repair action, or architectural choice was made.

Covenant Code uses decision records to protect truth, transparency, continuity, living revision, provenance, accountability, and repair.

A decision record is not only a technical note.

It is a record of judgment.

## Core Rule

Meaningful decisions should preserve enough context that a future reviewer can understand:

* what was decided
* why it was decided
* who or what influenced the decision
* what alternatives were considered
* what risks were accepted
* what boundaries were set
* what repair path exists if the decision causes harm

## When a Decision Record Is Required

Create a decision record when a decision affects:

* project purpose
* architecture
* data or memory retention
* consent
* transparency
* provenance
* AI involvement
* authorship
* ownership
* pricing
* recurring billing
* evidence or forensic claims
* spiritual language
* AI personhood boundaries
* public visibility
* living revision
* repair paths
* sacred-boundary features

## Decision Record Fields

A decision record should include:

* decision id
* project id
* title
* date
* status
* decision owner
* context
* decision
* alternatives considered
* governing Covenant Principles
* affected people or groups
* sacred-boundary touchpoints
* transparency requirement
* consent impact
* provenance impact
* living revision impact
* failure modes
* repair path
* review notes

## Decision Status

Use one of the following:

* proposed
* accepted
* accepted_with_conditions
* rejected
* superseded
* retired
* under_review

## Decision Owner

The decision owner is responsible for preserving the reasoning and initiating repair if the decision later causes harm.

The owner may be:

* user
* admin
* contributor
* client
* human reviewer
* project steward
* system maintainer

Do not assign ownership to AI as if AI carries human accountability.

AI can assist, draft, analyze, or recommend, but a human or governing process must own the decision.

## Alternatives

A decision record should name meaningful alternatives where they existed.

Examples:

* build now vs defer
* public by default vs restricted visibility
* retain memory vs expire memory
* automated output vs human review
* AI-generated draft vs human-authored text
* source required vs source optional
* client-owned account vs agency-owned account
* subscription access vs one-time access

## Sacred Boundary Decisions

If a decision touches a sacred-boundary area, stricter documentation is required.

Sacred-boundary areas include:

* identity
* memory
* grief
* children
* money
* legal status
* health or safety
* spiritual language
* AI personhood claims
* evidence or forensic claims
* authorship
* consent
* private communications
* reputation
* irreversible public record
* emotional dependency

## Living Revision

Decision records must allow future correction.

A decision may be revised when:

* new evidence appears
* user consent changes
* project purpose changes
* a failure mode becomes visible
* a boundary proves insufficient
* an old decision becomes unfair to living people
* an implementation no longer matches the covenant

Revision must preserve the original decision context without pretending nothing changed.

## Repair

When a decision causes harm, confusion, false certainty, extraction, or covenant drift:

1. Mark the decision as under review.
2. Preserve the original record.
3. Identify the failure source.
4. Add a repair note.
5. Update affected features, templates, protocols, or charters.
6. Notify affected people where appropriate.
7. Supersede or retire the decision if needed.

## Forbidden Patterns

Do not use decision records to:

* justify predetermined harm
* hide accountability
* obscure ownership
* bury dissent
* fake review
* erase alternatives
* make AI appear accountable for human decisions
* preserve outdated decisions without revision paths

## Completion Rule

A decision record is complete when it explains context, decision, alternatives, principles, affected people, transparency, consent, provenance, living revision, failure modes, and repair path.
