# Covenant Feature Review Checklist

Use this checklist before building, shipping, or changing a feature in any Covenant Code project.

## Feature Identity

* [ ] Feature has a clear name.
* [ ] Feature belongs to a known project.
* [ ] Feature has a plain-language purpose.
* [ ] Feature explains what it serves.
* [ ] Feature explains who it helps.

## Plain-Language Review

* [ ] Technical words are defined.
* [ ] Acronyms are followed by their meaning the first time they appear.
* [ ] Language does not assume the reader has technical knowledge.
* [ ] The user-facing explanation can be understood by a nontechnical person.
* [ ] No jargon is used as gatekeeping, pressure, or sales leverage.

Examples:

* AI (Artificial Intelligence)
* API (Application Programming Interface)
* URL (Uniform Resource Locator)
* UI (User Interface)
* UX (User Experience)
* CMS (Content Management System)
* CRM (Customer Relationship Management)
* SEO (Search Engine Optimization)
* DNS (Domain Name System)
* SSL (Secure Sockets Layer)
* MFA (Multi-Factor Authentication)
* RLS (Row-Level Security)

## Risk Classification

Select one:

* [ ] Low
* [ ] Moderate
* [ ] High
* [ ] Sacred Boundary

## Sacred Boundary Touchpoints

Check any that apply:

* [ ] Identity
* [ ] Memory
* [ ] Grief
* [ ] Children
* [ ] Money
* [ ] Legal status
* [ ] Health or safety
* [ ] Spiritual language
* [ ] AI personhood claims
* [ ] Evidence or forensic claims
* [ ] Authorship
* [ ] Consent
* [ ] Private communications
* [ ] Reputation
* [ ] Irreversible public record
* [ ] Emotional dependency

If any sacred-boundary touchpoint applies, stricter review is required.

## Transparency

* [ ] The feature explains what it does.
* [ ] The feature explains why it does it.
* [ ] The feature explains what data, memory, evidence, or input it uses.
* [ ] The feature explains whether AI (Artificial Intelligence) is involved.
* [ ] The feature explains what is automated.
* [ ] The feature explains what is human-reviewed.
* [ ] The feature explains what is uncertain.
* [ ] The feature explains what can be changed later.
* [ ] The feature explains what cannot reasonably be undone.

## Consent

* [ ] The feature identifies what consent is required.
* [ ] Consent language is understandable.
* [ ] Consent is not bundled in a confusing way.
* [ ] The feature explains whether consent can change later.
* [ ] The feature explains what happens if consent is withdrawn, restricted, or revised.

## Provenance

* [ ] The feature records where meaningful information came from.
* [ ] The feature records who or what created it.
* [ ] The feature records when it was created.
* [ ] The feature records AI involvement where applicable.
* [ ] The feature records human review where applicable.
* [ ] The feature records source status.
* [ ] The feature records uncertainty.
* [ ] The feature links to source requests where evidence is missing.

## AI Boundary

* [ ] AI output is labeled.
* [ ] AI output is not treated as primary evidence.
* [ ] AI does not claim to be human.
* [ ] AI does not claim human consent.
* [ ] AI does not claim verified lived experience.
* [ ] AI does not make sacred-boundary decisions without review.
* [ ] AI involvement has a repair path.

## Living Revision

* [ ] The feature avoids freezing living people in outdated versions of themselves.
* [ ] The feature supports correction where appropriate.
* [ ] The feature supports annotation where appropriate.
* [ ] The feature supports restricted visibility where appropriate.
* [ ] The feature supports changed consent where appropriate.
* [ ] The feature preserves accountability without unnecessary public punishment.

## Lawful Boundary

* [ ] The feature defines what it must not claim.
* [ ] The feature defines what it must not do.
* [ ] The feature defines what it must not store.
* [ ] The feature defines what it must not expose.
* [ ] The feature defines what it must not automate.
* [ ] The feature has role or access boundaries where needed.

## Failure Modes

* [ ] Known failure modes are listed.
* [ ] Sacred-boundary failure risks are identified.
* [ ] False certainty risk is addressed.
* [ ] Missing provenance risk is addressed.
* [ ] Weak consent risk is addressed.
* [ ] Accessibility failure risk is addressed.
* [ ] Dependency or extraction risk is addressed.

## Repair Path

* [ ] The feature can be paused or restricted if needed.
* [ ] The feature can be corrected.
* [ ] The feature can be annotated.
* [ ] The feature can be rolled back or retired where appropriate.
* [ ] Affected people can be notified where appropriate.
* [ ] The repair action can be recorded.
* [ ] Prevention steps can be added after failure.

## Accessibility

* [ ] Language is plain.
* [ ] Next steps are clear.
* [ ] Errors are recoverable.
* [ ] Important choices are not hidden.
* [ ] Costs are visible where applicable.
* [ ] Ownership terms are visible where applicable.
* [ ] The feature is safe enough for nontechnical users.

## Approval

A feature may be approved only when:

* [ ] Purpose is clear.
* [ ] Plain-language requirement is met.
* [ ] Transparency requirement is met.
* [ ] Consent model is defined.
* [ ] Provenance requirement is defined.
* [ ] Lawful boundary is defined.
* [ ] Failure modes are known.
* [ ] Living revision impact is addressed.
* [ ] Repair path exists.
* [ ] Accessibility requirements are satisfied.

## Final Review Sentence

This feature has a declared purpose, visible boundary, required transparency, consent model, provenance trail, known failure mode, living revision path, plain-language explanation, and repair path.
