# Project Charter Checklist

Use this checklist before adding a new project to Covenant Code Vessel.

## Project Identity

* [ ] Project has a clear name.
* [ ] Project has a plain-language purpose.
* [ ] Project has a vessel type.
* [ ] Project explains who it serves.
* [ ] Project explains what it must protect.
* [ ] Project explains what it must never do.

## Vessel Type

Select one:

* [ ] Signal vessel
* [ ] Continuity vessel
* [ ] Access vessel
* [ ] Service vessel
* [ ] Experimental vessel
* [ ] Governance vessel
* [ ] Memory vessel
* [ ] Other: [describe]

## Primary Principles

Select governing Covenant Principles:

* [ ] Truth
* [ ] Transparency
* [ ] Dignity
* [ ] Consent
* [ ] Continuity
* [ ] Living Revision
* [ ] Provenance
* [ ] Stewardship
* [ ] Repair
* [ ] Non-Extraction
* [ ] Lawful Boundary
* [ ] Accessibility
* [ ] Human Flourishing
* [ ] Signal Clarity
* [ ] Protective Privacy

## Plain-Language Requirement

* [ ] The charter does not assume technical knowledge.
* [ ] Acronyms are followed by their meaning the first time they appear.
* [ ] Technical words are defined in plain language.
* [ ] Language is not used as gatekeeping, pressure, or sales leverage.
* [ ] A nontechnical person can understand what the project does.

## Required Transparency

The charter explains:

* [ ] What the project does.
* [ ] What AI (Artificial Intelligence) involvement exists.
* [ ] What users, clients, visitors, or contributors are agreeing to.
* [ ] What data, memory, evidence, or money the project touches.
* [ ] What is automated.
* [ ] What is human-reviewed.
* [ ] What is uncertain.
* [ ] What can change later.
* [ ] What cannot reasonably be undone.
* [ ] What happens if something goes wrong.

## Sacred Boundary Touchpoints

Check any that apply:

* [ ] Identity
* [ ] Memory
* [ ] Grief
* [ ] Children
* [ ] Money
* [ ] Legal status
* [ ] Health or safety
* [ ] Spiritual language
* [ ] AI personhood claims
* [ ] Evidence or forensic claims
* [ ] Authorship
* [ ] Consent
* [ ] Private communications
* [ ] Reputation
* [ ] Irreversible public record
* [ ] Emotional dependency

## Required Boundaries

The charter defines what the project must not:

* [ ] Claim
* [ ] Do
* [ ] Store
* [ ] Expose
* [ ] Automate
* [ ] Sell
* [ ] Imply
* [ ] Preserve without purpose

## Provenance

The charter defines provenance requirements for:

* [ ] Claims
* [ ] Artifacts
* [ ] AI output
* [ ] Human review
* [ ] Client or user input
* [ ] Source material
* [ ] Decisions
* [ ] Revisions
* [ ] Repair actions

## Living Revision

The charter supports:

* [ ] Correction
* [ ] Annotation
* [ ] Changed consent
* [ ] Updated self-representation
* [ ] Restricted visibility
* [ ] Retirement
* [ ] Expiration where appropriate
* [ ] Accountability without unnecessary public punishment

## Repair Path

The charter defines how to repair:

* [ ] False claims
* [ ] Missing provenance
* [ ] Confusing language
* [ ] Consent problems
* [ ] Authorship issues
* [ ] Ownership issues
* [ ] Sacred-boundary harm
* [ ] Accessibility failures
* [ ] AI overreach
* [ ] Covenant drift

## Forbidden Uses

The charter clearly lists forbidden uses.

Forbidden uses should include anything that would violate:

* [ ] Truth
* [ ] Transparency
* [ ] Consent
* [ ] Dignity
* [ ] Lawful Boundary
* [ ] Repair
* [ ] Non-Extraction
* [ ] Living Revision
* [ ] Accessibility

## Completion Rule

A project charter is complete when it defines project purpose, vessel type, primary principles, sacred-boundary touchpoints, transparency requirements, plain-language requirements, provenance requirements, living revision support, repair path, and forbidden uses.

## Final Review Sentence

This project has a declared purpose, vessel type, visible boundaries, required transparency, provenance requirements, plain-language rule, living revision support, and repair path.
