# Repair Paths

## Purpose

Repair paths define how Covenant Code systems correct harm, confusion, false certainty, missing provenance, unclear consent, misattribution, exposure, extraction, inaccessible design, or covenant drift.

Repair is not an apology alone.

Repair is a structured response that preserves truth, protects people, corrects the record, and changes the system where needed.

## Core Rule

A feature is not complete until its repair path is known.

## Repair Principles

Repair must be:

* truthful
* transparent
* proportionate
* documented
* accessible
* protective where needed
* merciful with living people
* accountable without unnecessary exposure
* preventive, not only reactive

## Standard Repair Path

### 1. Pause

Stop or limit the harm where possible.

Examples:

* disable feature
* pause automation
* freeze artifact
* hide public output
* restrict visibility
* pause billing action
* stop publication
* prevent further sharing

### 2. Identify

Identify what failed.

Possible sources:

* unclear purpose
* missing transparency
* weak consent
* missing provenance
* false certainty
* broken boundary
* AI overreach
* human review failure
* inaccessible language
* unclear ownership
* outdated memory
* privacy used as concealment
* system bug
* bad assumption

### 3. Preserve

Preserve what is needed for accountability and repair.

Examples:

* audit record
* prior version
* source reference
* user report
* system log
* consent status
* reviewer notes
* affected artifact id
* correction request

Do not expose protected people unnecessarily.

### 4. Correct

Correct the user-facing issue.

Examples:

* edit claim
* update label
* add uncertainty
* correct attribution
* clarify AI involvement
* revise ownership terms
* add missing source status
* restrict visibility
* update consent
* roll back a change
* repair accessibility

### 5. Notify

Notify affected people where appropriate.

Good notification is:

* plain
* honest
* specific
* proportionate
* actionable
* not alarmist
* not evasive

### 6. Review

Determine whether the failure was isolated or systemic.

Ask:

* Did the design cause this?
* Did the language cause this?
* Did the AI prompt cause this?
* Did the consent flow fail?
* Did provenance fail?
* Did review fail?
* Did a boundary fail?
* Did a template cause repeated confusion?

### 7. Prevent

Change the system to reduce recurrence.

Examples:

* update copy
* add validation
* add review
* change permissions
* change defaults
* add audit fields
* improve accessibility
* add source requirements
* update templates
* update protocols
* retire the feature

## Repair Outcomes

Use one of the following outcomes:

* corrected
* corrected_with_notice
* restricted
* retired
* reverted
* refunded_or_compensated
* escalated
* unresolved
* unable_to_repair_fully
* preserved_for_future_review

## Repair by Failure Type

### False Claim

Repair by:

* relabeling evidence status
* adding uncertainty
* correcting the claim
* preserving prior version
* opening a source request
* notifying affected people where appropriate

### Missing Provenance

Repair by:

* marking provenance incomplete
* opening source request
* adding origin and review fields
* restricting unsupported claims
* updating provenance requirements

### Consent Failure

Repair by:

* pausing use
* clarifying consent
* restricting future use
* honoring withdrawal where possible
* preserving audit history
* updating consent flow

### Authorship Failure

Repair by:

* correcting attribution
* preserving correction history
* notifying affected contributors where appropriate
* updating contribution process
* restricting disputed artifacts

### Ownership Failure

Repair by:

* clarifying ownership terms
* correcting client-facing documents
* providing transfer or export path where appropriate
* updating contract or proposal templates
* preserving decision record

### AI Overreach

Repair by:

* labeling AI involvement
* adding human review
* restricting AI autonomy
* correcting overconfident claims
* updating prompts and validation
* documenting model limitations

### Living Revision Failure

Repair by:

* adding updated self-representation
* annotating outdated record
* restricting public visibility where appropriate
* preserving audit history without public punishment
* honoring changed consent where possible
* preventing old memory from being used as current truth

### Accessibility Failure

Repair by:

* rewriting in plain language
* improving interface readability
* adding recovery steps
* clarifying next actions
* reducing jargon
* improving disability access
* testing with non-expert users where possible

### Extraction Failure

Repair by:

* clarifying exchange
* correcting ownership and cost terms
* offering opt-out, refund, transfer, or cancellation where appropriate
* removing dependency traps
* updating public language and contracts

## Sacred-Boundary Repair

Sacred-boundary repair requires stricter care.

When repair touches identity, memory, grief, children, money, legal status, health or safety, spiritual language, AI personhood claims, evidence, authorship, consent, private communications, reputation, irreversible public record, or emotional dependency, identify:

* affected people
* consent issue
* provenance issue
* public visibility
* private harm
* boundary breached
* notification requirement
* living revision impact
* prevention requirement
* reviewer

## Forbidden Repair Patterns

Do not repair by:

* silently rewriting history
* deleting accountability
* blaming the user for unclear design
* hiding system failures
* exposing protected people unnecessarily
* using privacy to conceal exploitation
* using truth to punish transformation
* issuing vague apologies without structural change
* continuing harmful features unchanged
* preserving harmful public memory where restriction is warranted

## Completion Rule

A repair path is complete when it defines how to pause, identify, preserve, correct, notify, review, prevent, and document the issue.

If full repair is impossible, the remaining harm or uncertainty must be documented.
