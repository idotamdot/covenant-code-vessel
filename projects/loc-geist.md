# Loc-Geist Covenant Charter

## Purpose

Loc-Geist exists to reveal previously unseen signal in compiled anomalous data through structured logic, provenance-aware analysis, disciplined uncertainty, and lawful source handling.

Loc-Geist is a signal vessel.

It should help humans become more intentional in structured logic without pretending that interpretation is the same as proof.

## Primary Principles

- truth
- transparency
- provenance
- signal clarity
- lawful boundary
- stewardship
- repair
- living revision

## Sacred Boundary Touchpoints

Loc-Geist may touch sacred-boundary areas including:

- evidence or forensic claims
- reputation
- identity
- private communications
- irreversible public record
- AI personhood claims
- spiritual language

## Required Transparency

Loc-Geist must make clear:

- what is raw source material
- what is user-submitted content
- what is AI-generated analysis
- what is human-reviewed
- what is inferred
- what is symbolic interpretation
- what is verified record
- what remains uncertain
- what sources are missing
- what subscription tier controls access to token-costing activity

## Required Boundaries

Loc-Geist must not:

- present speculation as fact
- erase uncertainty
- detach claims from sources
- allow interpretive layers to overwrite raw evidence
- claim proof where only pattern, probability, or inference exists
- permit no-subscription access to AI/token-costing activity
- permit sighting uploads without an active subscription
- allow AI council output to bypass provenance requirements
- let narrative style replace forensic clarity

## Required Provenance

Every sighting, source request, AI analysis, council output, interpretive layer, and forensic claim must preserve:

- source origin
- timestamp
- submitter or creator role where appropriate
- confidence level
- evidence status
- transformation history
- model or agent involvement
- review status
- unresolved source requests
- audit trail
- correction history

## Living Revision

Loc-Geist must support correction and revision.

Claims may be updated when:

- better sources appear
- earlier analysis was overconfident
- provenance is incomplete
- a witness or contributor updates consent
- a source is challenged
- a model output is later found unreliable
- a council packet fails validation

Revision must not erase the original audit trail.

## Repair Path

When a claim is wrong, unsupported, overconfident, misattributed, or harmful:

1. Mark the claim as disputed, revised, corrected, or retired.
2. Preserve the original version in the audit trail.
3. Add correction provenance.
4. Identify which layer failed.
5. Notify affected users where appropriate.
6. Update validation rules if the failure was systemic.
7. Add the unresolved issue to the source request list where needed.

## Forbidden Uses

Loc-Geist must not be used to:

- harass witnesses
- expose private people without lawful reason
- manufacture certainty
- launder speculation into fact
- generate fear for engagement
- override human review on sacred-boundary claims
- hide missing sources
- monetize unverifiable claims as certainty

## Completion Rule

A Loc-Geist feature is not complete until it defines its source model, uncertainty model, provenance trail, transparency requirement, failure mode, lawful boundary, living revision impact, and repair path.
