# Feature Intake Protocol

## Purpose

The Feature Intake Protocol defines how new features enter a Covenant Code project.

Feature intake prevents random additions, scope drift, false urgency, hidden harm, unclear ownership, and features that do not serve the project covenant.

## Intake Rule

No feature should be built until it can answer:

What does this serve?

If the purpose is unclear, the feature is not ready.

## Required Intake Fields

Every feature request should include:

* feature name
* requesting person or source
* project
* purpose
* user-facing value
* affected people or groups
* data, memory, money, evidence, or identity touched
* AI involvement
* required transparency
* consent requirement
* provenance requirement
* lawful boundary
* known risk
* repair path
* accessibility needs
* success condition
* forbidden use

## Feature Purpose

A feature purpose should be written in plain language.

Poor purpose:

> Add AI automation.

Better purpose:

> Help a small-business owner understand which client follow-ups need attention without hiding the fact that AI is summarizing messages.

## Principle Mapping

Every feature should map to one or more Covenant Principles:

* truth
* transparency
* dignity
* consent
* continuity
* living revision
* provenance
* stewardship
* repair
* non-extraction
* lawful boundary
* accessibility
* human flourishing
* signal clarity
* protective privacy

If no principle applies, the feature should be challenged.

## Risk Classification

Classify the feature as:

* low
* moderate
* high
* sacred-boundary

## Low Risk

Low-risk features do not affect identity, money, memory, consent, private communications, evidence, spiritual language, or reputation.

## Moderate Risk

Moderate-risk features affect workflow, user expectation, visibility, billing context, automation, or saved preferences.

## High Risk

High-risk features affect user trust, payments, public claims, legal exposure, data sharing, system access, or important records.

## Sacred-Boundary Risk

Sacred-boundary features touch:

* identity
* memory
* grief
* children
* money
* legal status
* health or safety
* spiritual language
* AI personhood claims
* evidence or forensic claims
* authorship
* consent
* private communications
* reputation
* emotional dependency
* irreversible public record

## Intake Decision

After intake, the feature can be:

* accepted for design
* returned for clarification
* deferred
* rejected
* escalated to covenant review

## Rejection Reasons

A feature should be rejected if it requires:

* hidden extraction
* deceptive simulation
* false certainty
* unclear ownership
* unbounded memory
* no repair path
* user confusion as leverage
* privacy as concealment
* public permanence where living revision is required

## Completion Rule

Feature intake is complete when the feature has a purpose, principle mapping, risk level, transparency requirement, consent model, provenance requirement, lawful boundary, and repair path.
