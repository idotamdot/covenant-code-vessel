# Repair Loop Protocol

## Purpose

The Repair Loop Protocol defines how Covenant Code systems respond when something is wrong, harmful, unclear, misleading, broken, overconfident, inaccessible, misattributed, or misaligned with the project covenant.

Repair is not an afterthought.

Repair is part of the architecture.

## Core Rule

A system trusted with meaningful memory, money, identity, consent, authorship, evidence, or belief must be able to repair harm.

A feature with no repair path is not complete.

## When a Repair Loop Is Required

A repair loop is required when:

* a claim is wrong
* a claim is overconfident
* a source is missing or misrepresented
* consent was unclear or changed
* authorship was misattributed
* ownership terms were unclear
* a user was confused by system language
* an AI output was presented too strongly
* private information was exposed
* memory became unfair or outdated
* a person was frozen into an old version of themselves
* a client was misled about cost, scope, ownership, or AI involvement
* a sacred-boundary feature failed
* a system boundary was violated
* a project drifted from its covenant

## Repair Loop Stages

### 1. Pause

Stop the harmful, unclear, or disputed behavior where possible.

This may include:

* disabling a feature
* hiding a public artifact
* freezing an artifact
* pausing a workflow
* suspending automation
* blocking publication
* restricting visibility
* pausing billing-related action

### 2. Identify

Identify what failed.

Possible failure sources include:

* unclear purpose
* missing transparency
* weak consent
* missing provenance
* incorrect source
* AI overreach
* human review failure
* inaccessible language
* unclear ownership
* broken boundary
* system bug
* bad assumption
* outdated memory
* unreviewed sacred-boundary impact

### 3. Preserve

Preserve enough record truth to support accountability and repair.

Preservation may include:

* audit record
* prior version
* source snapshot
* consent status
* user report
* reviewer notes
* system logs
* correction request
* affected artifact id

Preserve carefully.

Do not expose protected people unnecessarily.

### 4. Correct

Correct the user-facing issue.

Correction may include:

* editing a claim
* changing a label
* adding uncertainty
* correcting attribution
* clarifying AI involvement
* revising ownership terms
* adding missing source status
* restricting visibility
* updating consent
* updating pricing language
* rolling back a change

### 5. Notify

Notify affected people where appropriate.

Notification should be clear, plain, and proportionate.

Do not create unnecessary alarm, but do not hide meaningful harm.

### 6. Review

Review whether the failure was isolated or systemic.

Ask:

* Did the feature design cause this?
* Did the language cause this?
* Did the AI prompt cause this?
* Did the consent model fail?
* Did provenance fail?
* Did review fail?
* Did a boundary fail?
* Did a template cause repeated confusion?

### 7. Prevent

Update the system to reduce recurrence.

Prevention may include:

* changing copy
* adding validation
* adding review
* adding audit fields
* changing permissions
* changing defaults
* improving accessibility
* adding source requirements
* updating templates
* updating protocols
* retiring the feature

## Repair Outcomes

A repair loop may end with:

* corrected
* corrected with notice
* restricted
* retired
* reverted
* refunded or compensated where appropriate
* escalated
* unresolved
* unable to repair fully
* preserved for future review

## Living Revision Repair

When repair involves a living person, avoid treating old information as the whole truth.

Support:

* correction
* annotation
* restricted visibility
* updated self-representation
* changed consent
* expiration
* forgiveness where appropriate
* audit preservation without public punishment

## Sacred Boundary Repair

When repair touches a sacred-boundary area, require stricter care.

Sacred-boundary repair should identify:

* affected people
* boundary breached
* consent issue
* provenance issue
* public visibility
* private harm
* emotional dependency risk
* living revision impact
* required notification
* future prevention

## Forbidden Repair Patterns

Do not repair by:

* silently rewriting history
* deleting accountability
* blaming the user for unclear design
* hiding system failures
* preserving harmful public memory unnecessarily
* exposing protected people to prove transparency
* using privacy to conceal exploitation
* using truth to punish transformation
* issuing vague apologies without structural change
* continuing the harmful feature unchanged

## Completion Rule

A repair loop is complete when the issue has been paused where needed, identified, preserved, corrected, reviewed, and prevented where possible.

If full repair is impossible, the remaining harm or uncertainty must be documented.
