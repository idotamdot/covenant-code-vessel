# Covenant Review Protocol

## Purpose

The Covenant Review Protocol defines how a feature, project, artifact, workflow, or system change is reviewed before it is considered complete.

Covenant review is not only a technical review.

It is a purpose, boundary, transparency, consent, provenance, living revision, and repair review.

## When Review Is Required

A covenant review is required when a change touches:

* identity
* memory
* consent
* authorship
* money
* legal status
* private communications
* spiritual language
* AI-generated claims
* AI personhood claims
* evidence or forensic claims
* reputation
* children
* grief
* emotional dependency
* irreversible public record
* client ownership
* recurring billing
* user data
* source provenance

## Review Inputs

Before review, the builder must provide:

* feature name
* project name
* purpose
* affected people or groups
* governing Covenant Principles
* expected user-visible behavior
* AI involvement
* data or memory touched
* consent model
* transparency requirement
* provenance requirement
* lawful boundary
* known failure mode
* living revision impact
* repair path
* accessibility considerations
* forbidden uses

## Review Questions

For every reviewed change, answer:

1. What does this serve?
2. Who could be affected?
3. What must be transparent?
4. What consent is required?
5. Can consent change later?
6. What is recorded?
7. What should not be recorded?
8. What provenance is required?
9. What boundary prevents misuse?
10. What could fail?
11. How is harm repaired?
12. Does this clarify signal or create noise?
13. Could this unfairly freeze someone in an outdated version of themselves?
14. Is privacy being used as protection or concealment?
15. Is the system creating dependency through confusion?

## Review Outcomes

A covenant review may result in one of the following outcomes:

* approved
* approved with conditions
* needs revision
* blocked
* retired

## Approved

A feature is approved when its purpose, transparency requirement, consent model, provenance trail, lawful boundary, living revision impact, failure mode, and repair path are clear.

## Approved With Conditions

A feature may be approved with conditions when it is acceptable only if specific requirements are added before release.

Conditions may include:

* clearer user-facing language
* consent change support
* provenance logging
* audit trail updates
* stricter role permissions
* accessibility improvements
* source labeling
* repair path documentation
* human review before publication

## Needs Revision

A feature needs revision when its purpose is valid but its design is incomplete, unclear, unsafe, inaccessible, or insufficiently transparent.

## Blocked

A feature is blocked when it depends on:

* deception
* hidden extraction
* false certainty
* manipulative dependency
* unclear ownership
* unrepairable harm
* spiritual manipulation
* deceptive AI personhood claims
* privacy language used to hide accountability
* permanent public memory where living revision is required

## Retired

A feature is retired when it no longer serves the covenant, cannot be repaired, or has been replaced by a safer design.

## Sacred Boundary Escalation

If a feature is classified as sacred-boundary, it must receive stricter review.

Sacred-boundary review requires:

* explicit purpose
* explicit user-facing transparency
* explicit consent model
* provenance trail
* role or access boundary
* failure mode
* repair path
* living revision assessment
* human review where appropriate

## Completion Rule

A covenant review is complete only when the reviewer can state:

This feature has a declared purpose, visible boundary, required transparency, consent model, provenance trail, known failure mode, living revision path, and repair path.
